## 技术总结

### 架构决策

| 决策     | 选择                 | 理由                                 | 效果                         |
| -------- | -------------------- | ------------------------------------ | ---------------------------- |
| 桌面框架 | Electron 30          | 跨平台 + Web 技术栈                  | 开发效率高，一套代码多平台   |
| UI 框架  | Naive UI             | 暗色主题内置 + TypeScript + 丰富组件 | 开箱即用，零配置暗色         |
| 图片引擎 | Sharp                | 高性能 C++ 级图片处理                | 压缩/转换速度快              |
| 构建工具 | vite-plugin-electron | Vite HMR + Electron 一体             | 开发体验极佳                 |
| IPC 架构 | 模块化 handler       | 每个功能独立文件                     | 解耦清晰，易维护             |
| 状态传递 | provide/inject       | 轻量级全局状态                       | 无额外依赖（可选升级 Pinia） |

### 技术挑战

| 挑战                         | 解决方案                            | 教训                                                       |
| ---------------------------- | ----------------------------------- | ---------------------------------------------------------- |
| `CompressOutline` 图标不存在 | 改用 `ContractOutline`              | 使用第三方图标库前需确认图标名是否存在                     |
| `useMessage()` 静默失败      | 添加 `NMessageProvider` 包裹        | Naive UI 的 `useMessage` 必须有 Provider 祖先              |
| ESM 中 `require()` 不可用    | 改为 `import sharp from 'sharp'`    | ESM 项目统一用 import，不混用 require                      |
| Sharp 原生模块打包失败       | `rollupOptions.external: ['sharp']` | 原生 .node 模块必须外部化，不能被 Rollup 打包              |
| `removeListener` 未暴露      | 改用 preload 暴露的 `off()`         | 严格使用 preload 暴露的 API，不要假设 Node.js API 完整可用 |

## 经验沉淀

### 最佳实践

1. **IPC 模块化**：每个功能域一个 `xxx.handler.ts`，主进程只做注册，职责分离清晰
2. **纯逻辑提取**：将业务逻辑（颜色替换、质量推荐、大小计算）提取到 `core/*.utils.ts`，便于单元测试
3. **ESM 统一**：Electron 主进程全部用 `import` 而非 `require`，避免混用导致运行时错误
4. **原生模块外部化**：Sharp 等含 `.node` 二进制的包必须在 Rollup external 中声明
5. **preload API 对齐**：渲染进程严格使用 preload 暴露的方法（`on`/`off`/`invoke`），不假设完整 Node API

### 改进建议

1. **Pinia 状态管理**：当前用 `ref` 局部状态 + `provide/inject`，复杂度增长后建议引入 Pinia
2. **Worker 线程池**：大文件批量压缩/转换时，Sharp 运算应放入 Worker 线程避免阻塞主进程
3. **Spectron/Playwright Electron**：使用 Playwright 的 Electron 支持进行端到端测试，解决 IPC 用例 BLOCKED 问题
4. **错误边界**：添加 Vue `errorCaptured` 全局错误边界 + 主进程 `uncaughtException` 处理
5. **国际化**：当前硬编码中文，后续可引入 vue-i18n 多语言
