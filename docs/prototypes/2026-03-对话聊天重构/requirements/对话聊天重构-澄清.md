## 原始需求

```
基于 Chater.vue 及相关组件的代码分析，制定重构方案。
原则：原代码保留不删除，新建独立文件处理，建立清晰注释和稳健架构。

核心内容：
- 将 Chater.vue（902行）和 chat-right-show.vue（1109行）巨石组件拆分
- 按关注点分离为独立模块（composables / components / types / api / store）
- 完善 TypeScript 类型定义
- 统一中文注释规范
- 完善错误处理（仅输出控制台）
- 保持原有业务逻辑不变
- 路由可回退方案
```

## 澄清问答

### Q1: 子组件复用范围

重构方案提到"子组件优先复用"，现有 `components/chater/` 下有 **20 个子组件**（chat-audio、chat-avatar、chat-block、chat-channel、chat-comment、chat-cover、chat-customer、chat-document、chat-factor-provider、chat-factor、chat-image、chat-infor、chat-messager、chat-search、chat-send-mms、chat-sessioner、chat-texter、chat-video、data.tsx 等）。

- 哪些子组件在本次重构中**直接复用不修改**？
- 哪些子组件需要**适配新架构**（如 props/emit 接口调整）？
- `data.tsx` 是否需要迁移或重构？

**A**: 所以的都直接复用，不需要修改。可以适配新的架构。但要做隔离，不要影响旧的逻辑。我需要无缝切换新旧版本。

### Q2: WebSocket 消息格式与 cmd 类型完整性

重构方案中 `WsCommand` 枚举定义了 `LOGIN(0)、HEARTBEAT(1)、OFFLINE(2)、UPLINK(3)、DOWNLINK(4)、CONVERSATION(6)`。但在 `Chater.vue` 的 `messageHandlers` 中只处理了 cmd=3、4、6。

- cmd=0（LOGIN）、cmd=1（HEARTBEAT）、cmd=2（OFFLINE）是否也需要在 `useChatSocket` 中处理？
- 是否还有其他未在代码中体现的 cmd 类型？
- 后端是否有 WebSocket 协议文档可供参考？

**A**:

### Q3: 旧版 `axios.CancelToken` 替换为 `AbortController` 的影响范围

方案提到用 `AbortController` 替换已废弃的 `axios.CancelToken`。当前 `chat-right-show.vue` 的 `initChat` 方法通过 `cancelTokenSource` 从父组件传入进行请求取消。

- 项目使用的 axios 版本是否支持 `AbortController`（需 axios ≥ 0.22.0）？
- 替换 `CancelToken` 时，是否需要同时修改 `utils-request.ts` 中的请求拦截器？
- 除了聊天模块，其他模块是否也在使用 `CancelToken`，是否需要统一替换？

**A**:

### Q4: Store 数据迁移策略

现有 `general.ts` 中的 `userChat` Store（命名为 `users`）管理着消息缓存。重构提议新建 `store/chat.ts`。

- 新旧 Store 是否需要**共存**一段时间（因为旧 `Chater.vue` 仍保留使用）？
- 如果共存，两个 Store 是否存在数据冲突风险（如同一 conversationId 的缓存）？
- 缓存上限目前为 100 条，是否需要在新 Store 中调整？

**A**:

### Q5: 错误处理策略细化

方案要求"所有错误仅输出到控制台，不弹窗提示"。但当前 `chat-right-show.vue` 中：

- 发送消息失败时调用了 `createNotice` 弹窗提示用户（特别是 `error.code == 10038` 的情况）
- 加载历史消息失败时也有弹窗提示
- **是否真的所有场景都不弹窗？** 用户发送消息失败时不提示是否会影响体验？
- 是否应区分：**用户操作型错误**（如发送失败 → 需提示）vs **系统后台型错误**（如轮询失败 → 仅控制台）？

**A**:

### Q6: 路由切换机制

方案中采用注释切换路由的方式回退：

```typescript
component: () => import('@/views/chat/index.vue'),   // 新版
// component: () => import('@/views/Chater.vue'),     // 旧版
```

- 是否考虑使用**环境变量/Feature Flag** 来控制新旧版本切换，而非手动注释代码？
- 是否需要支持**同一用户在新旧版本间对比测试**？
- 切换到新版后，旧版 `Chater.vue` 的保留期限是多久？

- 重构期间如果有紧急 bugfix 需要上线，如何处理分支合并冲突？

### Q8: 性能优化优先级

方案提到了多项性能问题（手动队列、computed 排序、定时器管理），但未明确优先级。

- `sortedContactItems` 每次重新排序整个列表 — 是否在当前数据量下已造成性能瓶颈？列表的典型大小是多少？
- `processQueue` 手写任务队列 — 是否已出现过消息丢失或乱序问题？
- 这些性能优化是**本次重构必须完成**的，还是可以在后续迭代中完善？

- `sendType` 字段的可能值有哪些？当前代码中 `sendType != 2` 有特殊逻辑处理。
