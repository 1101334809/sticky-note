## 需求背景

用户需要一款轻量级桌面便签工具，用于随时记录灵感、待办和备忘事项。市面上的便签应用要么过于简陋（功能不足），要么过于臃肿（资源占用高）。本项目旨在打造一款**结构清晰、拓展性强**的 Electron 桌面便签，当前全部功能在本地运行，但架构层面预留远程 API 对接能力，确保未来可无缝迁移到云端同步模式。

## 目标用户

| 用户角色       | 特征                         | 使用场景                               |
| -------------- | ---------------------------- | -------------------------------------- |
| 知识工作者     | 日常需要快速记录灵感和待办   | 工作中随手记录会议要点、灵感、临时任务 |
| 桌面便签爱好者 | 习惯在桌面放置便签备忘       | 将重要提醒钉在桌面，始终可见           |
| 开发者/设计师  | 需要轻量笔记工具，不依赖云端 | 记录代码片段、设计想法、bug 备忘       |

## 功能描述

### 核心功能

1. **便签创建与编辑**：用户创建便签后在独立窗口中编辑纯文本内容，内容自动保存（debounce 500ms），无需手动保存
2. **多窗口管理**：每个便签为独立窗口，支持自由拖拽移动、调整大小，窗口位置和尺寸持久化记忆
3. **自定义标题栏**：采用 frameless 窗口 + 自绘标题栏，包含 5 个快捷按钮：➕新建 / 🗑️删除 / 🎨颜色 / 📌置顶 / ❌关闭
4. **管理面板**：集中展示所有便签，支持卡片网格和列表两种视图切换，支持搜索和排序
5. **回收站**：删除便签进入回收站，支持恢复和永久删除
6. **颜色主题**：至少 6 种便签颜色（黄/绿/蓝/粉/紫/橙），便于视觉分类
7. **标签系统**：为便签添加标签（tag），管理面板按标签筛选
8. **系统托盘**：关闭主窗口最小化到托盘，右键菜单支持新建/显示/隐藏/退出
9. **窗口置顶**：便签可设为 always on top，确保重要信息始终可见
10. **快捷键**：全局快捷键新建便签（如 `Ctrl+Shift+N`），应用内快捷键关闭窗口（`Ctrl+W`）
11. **开机自启动**：默认开启，用户可在设置中取消
12. **数据导入/导出**：支持导出为 JSON、TXT 等格式，支持从文件导入便签
13. **自动更新**：应用内检测新版本并提示/自动更新

### 功能边界

- ✅ 包含：便签 CRUD、多窗口、本地存储、颜色标签、系统托盘、快捷键、导入导出、自动更新
- ✅ 架构预留：远程 API 对接（Repository 模式）、富文本编辑器升级、提醒功能、插件系统
- ❌ 不包含（当前版本）：云端同步、多设备协作、富文本编辑、提醒闹钟、插件市场

## 用户故事

### 便签操作

- 作为**用户**，我希望**快速创建一个新便签**，以便**随时记录突然出现的灵感或待办**
- 作为**用户**，我希望**编辑便签时内容自动保存**，以便**不用担心忘记保存导致内容丢失**
- 作为**用户**，我希望**自由移动和调整便签窗口大小**，以便**在桌面上合理安排便签的布局**
- 作为**用户**，我希望**为不同便签设置不同颜色**，以便**一眼区分工作、生活、紧急等不同类型**
- 作为**用户**，我希望**将重要便签置顶**，以便**确保关键信息始终显示在最前面**

### 管理操作

- 作为**用户**，我希望**在管理面板中查看所有便签**，以便**快速找到需要的内容**
- 作为**用户**，我希望**在卡片和列表视图之间切换**，以便**根据便签数量选择最舒适的浏览方式**
- 作为**用户**，我希望**搜索便签内容**，以便**在大量便签中快速定位**
- 作为**用户**，我希望**删除的便签进入回收站而不是永久删除**，以便**误删时可以恢复**
- 作为**用户**，我希望**为便签添加标签并按标签筛选**，以便**有序管理不同类别的便签**

### 系统集成

- 作为**用户**，我希望**应用开机自启动并恢复便签状态**，以便**无需每次手动打开**
- 作为**用户**，我希望**通过全局快捷键快速新建便签**，以便**在使用其他应用时也能即时记录**
- 作为**用户**，我希望**关闭主窗口时应用最小化到托盘**，以便**不占用任务栏但随时可用**
- 作为**用户**，我希望**导出便签数据**，以便**备份或迁移到其他设备**

## 业务规则

| 编号 | 规则                                         | 说明                             |
| ---- | -------------------------------------------- | -------------------------------- |
| BR01 | 便签内容变更后 500ms 内自动保存              | debounce 策略，避免频繁写入      |
| BR02 | 便签数量无上限                               | 不设人为限制，由系统资源自然约束 |
| BR03 | 删除便签进入回收站，回收站支持恢复和永久删除 | 回收站数据同样持久化存储         |
| BR04 | 便签窗口位置和大小在关闭后持久化             | 下次打开时恢复到上次的位置和尺寸 |
| BR05 | 开机自启动默认开启，用户可在设置中关闭       | 使用 Electron auto-launch API    |
| BR06 | 数据存储使用 JSON 文件                       | 简单可读，便于调试和迁移         |
| BR07 | 数据层采用 Repository 模式                   | 接口抽象，支持本地/远程无缝切换  |
| BR08 | 便签默认尺寸 300×300 像素                    | 用户可自由拖拽调整大小           |
| BR09 | 至少提供 6 种预定义便签颜色                  | 黄、绿、蓝、粉、紫、橙           |
| BR10 | 自定义标题栏包含 5 个快捷按钮                | 新建、删除、颜色、置顶、关闭     |

## 边界与异常

| 场景             | 触发条件                   | 处理方式                                 |
| ---------------- | -------------------------- | ---------------------------------------- |
| 大量便签         | 同时打开 20+ 便签窗口      | 给出性能提示，建议关闭不活跃便签         |
| 数据文件损坏     | JSON 文件格式异常          | 自动加载备份文件，提示用户数据已恢复     |
| 多显示器         | 便签窗口位于已拔除的显示器 | 检测屏幕边界，自动将窗口移回可见区域     |
| 存储空间不足     | 磁盘空间不足以保存数据     | 提示用户清理空间，保留最近一次有效的保存 |
| 导入数据格式错误 | 导入的 JSON 文件格式不匹配 | 提示格式错误，不覆盖现有数据             |
| macOS 权限限制   | 系统安全限制               | 引导用户到系统设置中授予权限             |

## 非功能需求

| 类型   | 要求                                                     |
| ------ | -------------------------------------------------------- |
| 性能   | 应用启动 < 2s，创建便签 < 1s，10 个窗口内存 < 300MB      |
| 存储   | JSON 文件存储，单文件管理，预留 Repository 抽象层        |
| 兼容   | Windows 10/11 + macOS 12+                                |
| 设计   | 现代扁平风格（类 Windows 11 Sticky Notes），自定义标题栏 |
| 安全   | 本地数据不加密（个人使用），预留加密接口                 |
| 可扩展 | Repository 模式支持远程切换、编辑器预留富文本升级接口    |
| 更新   | 集成自动更新，使用 electron-updater                      |

## 技术要点

### 架构分层

| 层级       | 目录                     | 职责                                    |
| ---------- | ------------------------ | --------------------------------------- |
| 窗口管理层 | `electron/windows/`      | 管理所有 BrowserWindow 的生命周期       |
| 业务服务层 | `electron/services/`     | 便签/标签/快捷键业务逻辑                |
| 数据仓库层 | `electron/repositories/` | Repository 接口 + 本地实现 + 远程预留   |
| 存储适配层 | `electron/storage/`      | LocalStorage（JSON）+ ApiClient（预留） |
| IPC 通信层 | `electron/ipc/`          | 主进程与渲染进程的通道定义              |
| 渲染页面层 | `src/`                   | Vue 3 + Pinia 渲染便签和管理面板        |

### 远程对接预留

当未来需要对接远程 API 时，仅需：

1. 实现 `RemoteNoteRepo`（实现 `INoteRepository` 接口）
2. 实现 `RemoteTagRepo`（实现 `ITagRepository` 接口）
3. 在 `RepoFactory` 中注册实现
4. 切换 `storageMode` 配置为 `'remote'`

**业务层代码零修改。**

详细接口规范、Repository 接口定义、同步策略参见 → [需求规格 · 第八章](./桌面便签-需求规格.md#八接口对接预留设计)
