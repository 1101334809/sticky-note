# 对话聊天模块重构方案

> 基于 `Chater.vue` 及相关组件的代码分析，制定本重构方案。
> 原则：**原代码保留不删除**，新建独立文件处理，建立清晰注释和稳健架构。

---

## 一、现状分析

### 1.1 核心文件清单

| 文件 | 行数 | 职责 |
|------|------|------|
| `views/Chater.vue` | 902 | 主页面：WebSocket连接 + 会话列表 + 搜索过滤 + 排序 + 置顶 + 定时器 + UI渲染 |
| `components/chater/chat-right-show.vue` | 1109 | 聊天窗口：消息展示 + 发送 + 缓存读写 + 端口状态轮询 + 新会话 + 彩信 |
| `utils/utils-websocket.tsx` | 214 | WebSocket封装：连接/重连/心跳/发送 |
| `store/general.ts` | 80 | Pinia Store：消息缓存管理 |
| `api/massTexting.ts` | 398 | API接口：聊天+群发+预约+AI机器人（混杂在一起） |
| `components/chater/*.vue` | ~20个 | 子组件：消息气泡、emoji、音频、视频等 |

### 1.2 核心问题

#### 🔴 架构层面

| 问题 | 影响 | 位置 |
|------|------|------|
| **单文件巨石组件** | `Chater.vue`（902行）和 `chat-right-show.vue`（1109行）承担过多职责，难以维护 | 两个主文件 |
| **业务逻辑与UI耦合** | WebSocket处理、数据过滤排序、定时器管理全部写在组件 `<script>` 内 | `Chater.vue` L1-625 |
| **API文件职责混乱** | 聊天相关API与群发、预约、AI机器人混在同一个 `massTexting.ts` 中 | `api/massTexting.ts` |
| **Store设计简陋** | `general.ts` 命名为 `users` 但实际是聊天缓存，无类型定义 | `store/general.ts` |

#### 🟡 代码质量

| 问题 | 示例 |
|------|------|
| **弱类型/无类型** | `ref<any[]>([])`、`ref<any>({})`、函数参数无类型注解 |
| **重复代码** | `twemojiParse()` 在 `Chater.vue` 和 `chat-right-show.vue` 中完全相同 |
| **清理逻辑重复** | `onUnmounted`、`onDeactivated`、`beforeunload` 三处几乎相同的清理代码 |
| **注释不一致** | 中英文混杂，大量注释掉的代码未清理 |
| **组件命名不规范** | 文件名 `Chater.vue` 拼写不标准（应为 `Chatter` 或 `Chat`） |

#### 🟠 功能/性能

| 问题 | 影响 |
|------|------|
| **手动队列实现** | `processQueue` 手写任务队列（L159-170），逻辑脆弱 |
| **axios直接创建CancelToken** | 使用已废弃的 `axios.CancelToken`（应改用 `AbortController`） |
| **定时器管理分散** | 5分钟批量刷新、30秒端口状态轮询、50ms缓冲定时器交叉管理 |
| **会话排序在computed中** | `sortedContactItems` 每次都重新排序整个列表 |

## 二、重构目标

1. ✅ **原代码保留**：`Chater.vue` 重命名为 `Chater.legacy.vue`，新代码独立文件
2. ✅ **模块解耦**：按关注点分离为独立模块
3. ✅ **类型安全**：完整的 TypeScript 类型定义
4. ✅ **清晰注释**：统一中文注释规范，说明代码作用
5. ✅ **稳健架构**：便于后续维护和扩展
6. ✅ **错误处理**：完善的错误处理，仅输出到控制台
7. ✅ **不变更不合理逻辑**：保持原有业务逻辑不变

## 六、关键改进点

### 6.1 废弃 API 替换

```diff
- // 旧：使用已废弃的 axios.CancelToken
- cancelTokenSource.value = axios.CancelToken.source()
+ // 新：使用原生 AbortController
+ const controller = new AbortController()
+ const signal = controller.signal
```

### 6.2 清理逻辑统一

```diff
- // 旧：三处重复的清理代码
- onUnmounted(() => { disconnect(); clearInterval(timers.value); ... })
- onDeactivated(() => { disconnect(); clearInterval(timers.value); ... })
- window.addEventListener('beforeunload', () => { disconnect(); ... })

+ // 新：统一的清理管理
+ const { registerCleanup } = useTimerManager()
+ registerCleanup(() => disconnect())
+ registerCleanup(() => clearInterval(timerId))
+ // onUnmounted 自动调用所有注册的清理函数
```

### 6.3 错误处理规范

```typescript
// 所有错误仅输出到控制台，不弹窗提示（符合需求第6条）
```
